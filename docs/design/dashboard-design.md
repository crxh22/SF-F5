## 1. Stack decision

**In-process, not a service.** `dashboard.py` is a new module of `sf_factory`; a `DashboardServer` instance is owned by the `Scheduler` and started as an asyncio task inside `run_forever`'s `TaskGroup` (control-plane §1 boundary: the dashboard backend *is the orchestrator's surface*). No second process, no reverse proxy, no new infrastructure class (DoD §9).

**HTTP mechanics: stdlib `http.server.ThreadingHTTPServer` worker threads, supervised by the asyncio task.** The `serve()` task runs the (already-bound, §6 `start()`) server on one daemon thread (thread-per-connection, `daemon_threads=True` **and `block_on_close=False`** — `ThreadingMixIn`'s default joins every handler thread in `server_close()`, so one hung client, legitimately alive up to `read_timeout_s`, would stall shutdown/restart for up to ~10s; daemon threads die with the process and the per-socket timeout bounds them anyway — per-socket `read_timeout_s`), and holds until cancelled; on `CancelledError`, `shutdown()`/`server_close()` run via `asyncio.to_thread` so the loop never blocks on the close path. Justification against the alternatives:
- *Hand-rolled HTTP over `asyncio.start_server`*: rejected — hand-parsing HTTP on the single most founder-critical surface is gratuitous correctness risk; `http.server`'s parser is battle-tested stdlib.
- *aiohttp / starlette*: rejected — a whole new dependency class (plus transitive deps) for one server-rendered page violates "minimal operational footprint" (DoD §9) when stdlib suffices.
- *Single-threaded `http.server`*: rejected — one hung phone connection would serialize all requests.
- Threads keep **all founder-client network I/O off the event loop by construction** — the same posture `notify.py` already takes for ntfy (§7 invariant 2). **Zero new dependencies.**

**Read path (GET):** handler threads NEVER touch the orchestrator's rw `Database`. Each GET opens its own short-lived `mode=ro` connection — the read access control-plane §2 explicitly sanctions for "the dashboard backend" — plus the liveness file mtime and artifact files (worktree path → `git cat-file <commit>:<path>` → integration branch, the `verify_integrity` precedence). GET handlers never write the DB (render errors are shown explicitly, logged to stderr; no off-loop event inserts).

**Write path:** exactly one **decision/state** write path (§3) — `POST /decision/<id>/answer`. Stated precisely because it is the invariant a reviewer must check (control-plane §1 "exactly one write path", amended by D-0016 to "read views + the single decision-answer endpoint + orchestrator-spawned Decision Sessions"): `POST /decision/<id>/session/message` is also write-bearing, but produces only an uncommitted, unregistered transcript file (inert for `verify_integrity`) plus the standard runner bookkeeping rows every `run_agent` call makes (process_registry, token_ledger, usage events) — never decision state, which only §3 writes. Handler threads marshal onto the orchestrator loop via `asyncio.run_coroutine_threadsafe`; **every thread→loop marshal is bounded** (POSTs by `dashboard.answer_timeout_s`, GET snapshot/poll marshals by `dashboard.read_timeout_s` — §6), expiry → 504 in Romanian.

**Page:** server-rendered single HTML page, inline CSS, **zero JavaScript on the main page**, `<meta http-equiv="refresh" content="{dashboard.refresh_s}">`. Only the Decision-Session page carries one small inline vanilla `<script>` (fetch-poll, §4) — no build pipeline, no SPA, no websockets (DoD §14); with JS off the session page still works by manual reload + form POST. **DoD §14's closest clause addressed head-on** ("any session UI richer than the inline decision-card input" is excluded): the card's inline input IS the session entry (DoD §9: opens "right in the card"); the session page is that same input's full-transcript rendering — a JS-free server-rendered reload page is the §14 baseline, and its poll script is a convenience inside §14's own "refresh/poll is enough" allowance, not a richer session UI (no websockets, no chat-app widgetry). This one-line deviation rationale must be recorded in the decision-log entry that ratifies this design.

**Bind:** `founder_channel.dashboard.bind: tailscale` → resolve at startup via `tailscale ip -4` (first address); empty output / nonzero exit / missing binary → `FactoryError`, **orchestrator start aborts** (a factory whose human gates cannot be answered breaches DoD §9 — fail at start, loudly, not at the first decision). The abort is **mechanically guaranteed by an eager first bind**: `cli run` calls `DashboardServer.start()` (resolve + bind, §6) *before* `run_forever` — never inside the §7 contain-and-restart supervisor, which would otherwise convert the promised hard abort into pipeline work running founder-less behind a 30s retry loop; only post-`start()` crashes are supervised (§7 row 1). Any other `bind` value is used literally (tests bind `127.0.0.1`). Bind to `(resolved_ip, dashboard.port)` only — never `0.0.0.0`. The tailnet IS the auth boundary (DoD §14): no login, no TLS, no multi-user. URLs carry **only numeric internal ids** (`/decision/3`, `/artifact/42`) — never artifact paths, content, or secrets; founder text travels only in POST bodies. Defense-in-depth headers on every response: `Content-Security-Policy: default-src 'none'; style-src 'unsafe-inline'; form-action 'self'; base-uri 'none'; frame-ancestors 'none'` (`form-action`/`base-uri`/`frame-ancestors` do **not** fall back to `default-src` — left unstated they are unrestricted), plus on the session page only `script-src 'nonce-…'; connect-src 'self'` — `fetch()`/XHR are governed by `connect-src`, which *does* fall back to `default-src 'none'`: without it the §4 poll script is dead on arrival and the page silently degrades to reload-only, invisible to headless tests (a unit test therefore asserts the session-page CSP contains `connect-src 'self'`). `X-Content-Type-Options: nosniff` everywhere.

**Routes** (everything else → 404 in Romanian): `GET /` (the page; ntfy deep links land on anchors `#decision/<id>` — fragment format pinned by `notify.dashboard_link`) · `GET /artifact/<ref_id>` (escaped read-only artifact view) · `GET /decision/<id>/session` · `GET /decision/<id>/session/poll?after=<n>` (JSON) · `POST /decision/<id>/answer` · `POST /decision/<id>/session/message`. POST bodies bounded by `dashboard.max_request_bytes`.

## 2. The three sections (DoD §9, verbatim)

**(a) Decizii așteptate** — one card per `db.pending_decisions()` row, anchor `id="decision/<id>"`. Rendering is **per-card contained**: a card whose render raises becomes an explicit Romanian error card (visible defect, same posture as R2's missing-gloss marker) — one bad row must never 500 the page and leave every *other* pending decision unanswerable from the phone, the exact DoD §13 "founder channel failing" condition. Card renders, maximally clear:
- Title: `Decizia #<id> — <unit name> (<unit id>)` + gate-kind gloss in Romanian (`critical_stage` → „etapă critică — aprobare necesară", `phase_signoff` → „semnătură de fază", `escalation_tradeoff` → „compromis de produs la escaladare", `business` → „decizie de business" — the full DDL `gate_kind` set, glossed even where no executor consumes it yet), created/age in founder format (§5).
- The **full decision-request artifact** (resolved via `request_artifact_id` → `artifact_refs` → file), escaped, structure preserved (§5 R5): question, options with trade-offs, recommendation, links. The founder protocol binds this content for EVERY gate, not only future architect-authored requests (DoD §9) — and today all three gates the implemented control plane can fire produce **control-plane-generated** requests: English machine templates with bare tokens for `critical_stage` (`_enter_awaiting_human`) and `phase_signoff` (`_enter_signoff`), and for `escalation_tradeoff` no request document at all (`request_artifact_id` anchored to the raw escalation payload, or even the spec). This build therefore **re-authors the three control-plane request paths in Romanian** (§9 scheduler.py delta): question + glossed unit/gate context + each declared option with its one-line consequence. `critical_stage`/`phase_signoff` templates carry a mechanical `Recomandare: approved` line — justified, not fabricated (R3): those gates fire only after every validation/audit/merge gate passed, so the recommendation is derived from mechanical facts. `escalation_tradeoff` gains a minimal Romanian **request-wrapper artifact** (question + glossed options + `/artifact/<ref>` link to the escalation payload — the payload is linked, never inlined as the card body; **no** recommendation line: a genuine product trade-off whose options come from the agents' payload), written + committed before the recording tx (§7 step order) and registered `kind='decision_request'` as the request anchor. Architect-authored requests (DoD §9, Etapa 5) render through the same path unchanged.
- **Option buttons**: one per declared option (source of truth: `models.GATE_ANSWERS[(unit_level, gate_kind)]`, §6) — Romanian label + the internal token in small print (protocol §6: id always with its gloss). A `(unit_level, gate_kind)` with **no `GATE_ANSWERS` entry** (the DDL enumerates `business`, which no executor consumes yet) renders the card **without buttons** — artifact + session entry + an explicit RO notice naming `cli decide` as the answer path — never a `KeyError` killing the page; §3 step 1 returns `INVALID_OPTION` (zero writes) for any answer POSTed against it. The recommended option (when the artifact declares one — machine-readable line `Recomandare: <option-token>`; marker contract ratified with this design and used by the re-authored control-plane templates from this build, OPEN-D1 covers only its adoption in architect role prompts) gets a visually distinct „★ Recomandat" badge; absent line = no badge, never fabricated.
- **Mechanical artifact links**: the unit's latest registered artifacts (spec, validation report, escalation payload…) as `/artifact/<ref_id>` links labeled kind-gloss + filename. Links *inside* agent-written text stay escaped plain text — synthesizing hyperlinks from agent-controlled content is an injection surface.
- **Inline Decision Session entry**: a text input + „Discută înainte de a decide" opening `GET /decision/<id>/session` (§4). Answering = tapping an option button, or conversing in the session and then explicitly confirming an option — both land on the same single write path (§3).

**(b) Acum în lucru** — the health strip, top of page:
- **Puls orchestrator**: `process.liveness_file` mtime → „acum Xs · HH:MM, DD-MM-YYYY"; older than `founder_channel.watchdog.staleness_threshold_s` → red „posibil căzut" marker.
- **Faze**: each non-terminal phase: name, state gloss, progress „X din Y etape gata".
- **Coadă etape**: stages by scheduling category — running ones named with the pipeline step reached (state gloss), counts for waiting/runnable.
- **Buget**: per active stage `db.unit_token_total` vs `budgets.per_stage[risk_class]` („142.000 / 300.000 tokeni · 47%"), plus factory totals (tokens + `cost_usd` where reported, `estimated` rows marked „estimat").
- **Ultimul incident**: newest `events` row with `event_type ∈ INCIDENT_EVENT_TYPES` (frozen constant: `alert`, `declared_failure`, `contract_change_request`, `alert_delivery_failed`, `cp_breach_attempt`, `timeout`, `usage_missing`) — type gloss + unit name + when.

**(c) Plan & istoric** — phases in DAG order, each with progress and its stages grouped **Finalizate** (DONE) / **În lucru** (non-terminal, showing the pipeline step reached) / **Planificate** (PENDING; phases not yet ingested show the phase itself + a link to the registered plan artifact when present). Footer on the section: „Vedere generată din planurile în git + stările din baza de date — nu este sursă canonică" (Doctrine §9; statuses from DB, plan content linked, never duplicated).

## 3. The single write path — POST /decision/<id>/answer

Form field `option=<token>`. The handler thread does **no** DB work; it submits `DashboardServer.answer(request_id, option, via=…)` to the orchestrator loop via `run_coroutine_threadsafe` and waits ≤ `dashboard.answer_timeout_s`.

`answer()` runs **on the loop**, serialized by an internal `asyncio.Lock` (one answer write at a time; also serializes factory-repo commits in-process):
1. **Validate** (loop-side read): request exists, else `UNKNOWN_REQUEST` (404, RO). `status='pending'`, else `ALREADY_ANSWERED` → **explicit no-op** (200 page „Decizia a fost deja înregistrată — nicio modificare", zero writes — idempotent double-tap by construction; concurrent taps serialize on the lock, the loser re-validates and lands here). `option ∈ models.GATE_ANSWERS.get((unit_level, gate_kind), ())`, else `INVALID_OPTION` (400 listing the valid options in Romanian, zero writes) — an unmapped `(level, gate_kind)` (e.g. `business`) therefore yields `INVALID_OPTION` for every option, never a `KeyError`.
1a. **Quiesce the session first (race fix, D-0019):** inside the lock, set a per-request *answering* flag (`post_message` refuses with an RO notice while set), then cancel any in-flight agent turn task and **await its termination** (its `try/finally` appends the cancelled-turn notice) — only after that is the transcript byte-stable. Rationale: a turn appending to the transcript file inside step 2's `await commit_paths(...)` window makes `register_artifact` hash post-append bytes while `git_commit` pins the pre-append blob — a registered ref that resolves nowhere → `verify_integrity` aborts the next orchestrator start (reproduced by the slice verifier). The answer semantically ends the session; nothing may write the transcript after it.
2. **Artifact first, committed (D-0015 order, mirrored exactly):** write `decision-answer-<id>.md` under the **factory repo** unit dir (`artifacts.unit_artifact_dir(factory.home, level, unit_id)` — same location and renderer contract as `cli decide`); if a session transcript file exists for this request, include it in the same commit; `await worktrees.commit_paths(factory.home, …)` **before** any DB write. When `commit_paths` returns `None` (nothing to commit — the byte-identical file is already committed, i.e. the retry after a commit-succeeded/tx-failed crash), register the artifact with the factory repo's **current HEAD** (`git rev-parse HEAD`, the commit that already contains the identical bytes) — the same contract as `cli decide`'s `_commit_decision_answer`: D-0015 requires the answer registered *with* its commit, and a NULL-commit factory-repo ref weakens `verify_integrity` resolution. A concurrent `cli decide` racing the factory-repo commit hits the git index lock → `GitError` → explicit 500, no partial state (an uncommitted, unregistered file is inert for `verify_integrity`).
3. **ONE synchronous transaction** (§7: no await inside): `register_artifact(kind='decision_answer', repo='factory', git_commit=sha)` [+ `register_artifact(kind='transcript')` when present] + `db.answer_decision(request_id, option, answer_ref_id)` + `insert_event('decision_answered', actor='founder', payload={request_id, answer, via, transcript_artifact_id?})`. **Lost cross-process race handled here, not as a 500:** a `cli decide` that completes its entire commit+tx inside step 2's `await commit_paths` window passes step 1 cleanly and collides only with `answer_decision`'s frozen `WHERE status='pending'` guard (rowcount≠1 → `FactoryError`) — `answer()` catches it and maps it to `ALREADY_ANSWERED` (the step-1 no-op page); the in-process lock covers only dashboard racers, never a second OS process. The superseding `decision-answer-<id>.md` commit left on factory main is harmless: the registered ref pins the winner's sha, so `git cat-file` resolution and sha256 integrity are unaffected.
4. Respond: 303 back to `/#decision/<id>` context → confirmation visible. Future timeout → 504 RO „se procesează — reîncarcă pagina" (retry is safe: step 1 makes it a no-op).

**Unblocking is the scheduler's normal job, untouched:** the next tick's BLOCKED-snapshot (pending-decision count changed) re-dispatches the unit within `process.loop_tick_s`; the executor's `_step_awaiting_human`/`_step_awaiting_signoff` consumes the answered row. This is DoD §12.A4 end-to-end: ntfy push → phone → dashboard card → tap → stage resumes.

**Marshalling choice justified:** the dashboard's HTTP handlers run on worker threads while §7 requires every DB write on the loop thread — `run_coroutine_threadsafe` is the canonical thread→loop bridge and returns a waitable future, which the handler needs to render the real outcome. An `asyncio.Queue` drained by the scheduler tick was rejected: it adds up to `loop_tick_s` latency plus a reply-future anyway (same mechanism, more parts); bare `call_soon_threadsafe` was rejected because the operation is a coroutine (it awaits `commit_paths`).

## 4. Decision Sessions (OPEN-4, confirmed into this slice by D-0011/D-0016)

Bounded, orchestrator-mediated conversation with runner role **`decision_session`** — already listed in `canon.founder_facing_roles`, so `run_agent` injects doctrine + conventions + the founder protocol automatically (D-0009). Needs a `models.decision_session` route (config addition, §6).

**State & mechanics** — `DecisionSessionManager`, loop-confined (all mutable state on the loop; HTTP threads only marshal):
- Per `request_id` at most one session; per session at most one agent turn in flight. While an answer for the request is in progress (the §3.1a *answering* flag, set inside the answer lock), `post_message` refuses with an RO notice and any in-flight turn is cancelled-and-awaited **before** the transcript is committed (D-0019).
- `post_message(request_id, text)`: validate (request still pending; not busy; `max_turns`/`budget_tokens` not exhausted — else explicit RO refusal, never silently queued) → append the founder turn to the **transcript file** `decision-session-<request_id>.md` under the factory-repo unit dir (crash-durable immediately; committed later by the answer path) → spawn one turn task in the dashboard's own `TaskGroup` — **hosted inside `serve()`**, so a supervisor restart cancels in-flight turns cleanly (the `DashboardServer`/manager objects are constructed once in `cli run`; only the `serve()` task restarts). The turn task's teardown is **`try/finally`: on failure AND on cancellation** it clears the in-flight flag and appends a failed-turn notice to the transcript — manager state survives a `serve()` restart, so a cancelled turn that skipped its finally would wedge the session „busy" forever. Turn spawn: `runner.run_agent(role='decision_session', kind='agent', unit_level/unit_id = the decision's unit, timeout_s=decision_session.turn_timeout_s, cwd=.factory/sessions/<request_id>/)`. First turn's prompt = the decision-request artifact content + unit/gate metadata + founder message + a frame instructing: discuss options and trade-offs in the founder's terms, never claim to execute anything, the founder confirms only via the dashboard buttons. Later turns pass `resume_session=<last session_id>` (claude `--resume`) with just the new message — and because every exchange is already in the transcript file, a restart-lost CLI session is rebuilt by feeding the transcript back as context.
- **The session NEVER writes state**: it converses. "Agents never see the DB" (DoD §2.3) covers state rows, and the gitignored scratch `cwd` keeps incidental files non-canonical — but **cwd does not confine a `claude -p` agent**: it runs as the orchestrator's user inside the live factory repo, ambient `.claude` settings govern its tool grants, and it is fed founder free-text (an injection surface); an agent writing/staging canonical files is an incident class already on record (D-0015 provenance note). The no-write claim is therefore made **actually structural: the `decision_session` role spawns tools-off** — `ModelRoute` gains an additive `tools: Literal['all','none'] = 'all'` field (config.py) honored by the adapters (runner.py delta: claude → the CLI's tools-off flagset, expected `--disallowedTools` over the full tool set, exact flags verified against the installed CLI at build; stub ignores it), ratified as a small CCR alongside `GATE_ANSWERS` (§6); a unit test asserts every session-turn argv carries the tools-off flag. Without that flag, enforcement would be prompt-level only — stated, not assumed. The *manager* (control plane) writes the transcript; only the founder's explicit option tap — the same §3 endpoint, rendered as the confirm buttons on the session page — writes the decision.
- **Bounds** (config, §6): `max_turns`, `turn_timeout_s` (passed to `run_agent`), `budget_tokens` (sum of the session's `AgentResult` usage; exhausted → session locked with an explicit RO notice; option buttons keep working). No idle timer: an inactive session holds no process or loop resources, so `max_turns × turn budget` already bounds total cost. Turn usage lands in `token_ledger` under the decision's unit (visible in the §2b burn) — see OPEN-D4 for the `context_budget` interaction.
- **Transcript → artifact**: committed to the factory repo and registered (`kind='transcript'`, already in the §2 DDL kind list) inside the same §3 step-2 commit + step-3 transaction when the founder confirms. If the founder answers via emergency `cli decide` instead, the uncommitted transcript remains as visible untracked residue (inert for integrity — never registered); acceptable for the emergency-only path, noted for the operator.

**HTTP shape (poll-based, no websockets):** `GET /decision/<id>/session` server-renders the full transcript + input + confirm buttons (works JS-free via reload); its inline script POSTs `…/session/message` and polls `GET …/session/poll?after=<n>` every `decision_session.poll_s` while busy → JSON `{turns:[{n, author, text, at}], busy, locked, turns_left}`; the script inserts text via `textContent` only (never `innerHTML`).

## 5. Founder-protocol conformance — testable rules

- **R1 Romanian only:** every founder-visible literal lives in the module-level `RO` string table; tests assert templates reference only `RO` entries and `RO` values contain no English UI words (denylist spot-check).
- **R2 no bare internal ids:** every id rendered with its human label (`Fundație (foundation)`, `Decizia #3 — Etapa: Schema de bază`); every internal token (states, gate kinds, event types, risk classes, option tokens) rendered through the `GLOSS` table; closure test: `GLOSS` covers all `StageState`/`PhaseState`/`SchedCategory` members, `GATE_ANSWERS` tokens, `INCIDENT_EVENT_TYPES`, **the full DDL `gate_kind` set (`critical_stage`, `business`, `phase_signoff`, `escalation_tradeoff`) and the golden config's `risk_classes` keys** — this rule claims gate kinds and risk classes are glossed, so the closure test must cover both, not just enum members. Unknown token at runtime → rendered `<token> (etichetă lipsă)` — visible defect, page never dies on it.
- **R3 recommendation marked:** parsed mechanically from the artifact's `Recomandare:`/`Recommendation:` line matching a declared option token → „★ Recomandat" badge on that button; absent/unmatched → no badge (never invent a recommendation).
- **R4 dates:** all founder-facing timestamps via `fmt_founder_ts`: `DD-MM-YYYY HH:MM` in `factory.timezone_founder` (Europe/Chisinau, `zoneinfo`); numbers grouped Romanian-style (`300.000`). Machine timestamps stay ISO UTC internally (conventions.md).
- **R5 escaping — markdown rendered as plain-text-with-structure:** ALL artifact/agent/founder-echoed content passes through `esc()` (`html.escape`) into `<pre style="white-space:pre-wrap">`. Decision: do **not** interpret markdown. Justification: agent-produced content is untrusted input by definition; a markdown→HTML renderer adds a dependency plus a sanitizer obligation and an XSS surface, while escaped markdown (headings, bullets, fences) remains perfectly legible as structured text; CSP (§1) backstops. Hostile-content tests are mandatory (§8).
- **R6 options-with-recommendation format** (protocol §3): cards always present *prepared options with a recommendation*, never an open question; the free-text session input is explicitly framed as discussion („Discută…"), with confirmation always an explicit option tap.

```python
        # addition mid-build a contract change -> STOP, so this cheap gap is closed pre-build).
    async def serve(self) -> None:
        """start() if not yet bound (supervised restarts re-run it, re-resolving the tailscale IP), run the
        ThreadingHTTPServer on a daemon thread (daemon_threads=True, block_on_close=False), hold until
        cancelled; on CancelledError run shutdown()/server_close() via asyncio.to_thread (never block the
        loop); hosts the §4 session-turn TaskGroup, so cancellation tears down in-flight turns with it."""
    async def answer(self, request_id: int, option: str, *, via: str) -> AnswerResult:
        """THE single write path (§3): loop-confined, lock-serialized; validate -> answer artifact
        (+ transcript) committed to the factory repo (D-0015 order; commit_paths None -> rev-parse HEAD) ->
        ONE sync tx (register_artifact + answer_decision + 'decision_answered' event, actor='founder').
        Already answered = explicit no-op — incl. a lost cross-process race vs cli decide, caught at the
        tx's pending guard and mapped to ALREADY_ANSWERED (§3.3), never a 500."""
```

**Scheduler wiring (additive contract amendment, CCR-class — architect ratifies with this design):**
- `Scheduler.__init__(…, dashboard: DashboardServer | None = None)` — optional, default None (tests/`run_until_blocked` unaffected).
- `_run()`: when present, `tg.create_task(self._dashboard_supervisor())` — containment + restart loop (§7); the task is excluded from `self._tasks`/quiescence accounting and is **cancelled** on every `_run` exit path (else `run_until_blocked`'s TaskGroup would never close). **The supervisor contains ALL exceptions per iteration — nothing ever escapes into the TaskGroup**: in particular its own alert publish follows the control-plane §6 `NotifyError` contract (`alert_delivery_failed` event, never re-raise) — `NtfyPublisher.publish` raises on timeout/HTTP failure, so an unwrapped publish would tear down the whole orchestrator exactly when the founder channel is already down. **Paging is deduplicated**: an 'alert' event on every restart (audit trail, restart counter in the payload), but the max-priority page fires on the first crash and then at most every `dashboard.page_every_n_restarts`-th — a persistent 30s crash loop must not emit ~2,880 identical max-priority pages/day (alarm fatigue degrades the DoD §9 minimal-attention channel). Between restarts the supervisor re-checks the bind every `dashboard.bind_recheck_s` (§7 IP-drift row).
- `cli run` constructs `DashboardServer(cfg, db, runner, notify)` and passes it; `cli resume`/tests pass None unless exercising the dashboard.
- **`models.GATE_ANSWERS: Mapping[tuple[str, str], tuple[str, ...]]`** (additive amendment): the gate answer vocabulary, moved from `scheduler.py` privates — **not verbatim**: the signoff executor today also tolerates a `changes_requested` alias next to `changes` (scheduler.py `_step_awaiting_signoff`); the swap **drops the alias**, a deliberate behavioral edit inside the owned scheduler.py delta to the vocabulary `cli decide` free-text tolerated, so the one-source mapping and the pin test agree. Mapping: `('stage','critical_stage')`/`('stage','escalation_tradeoff')` → `('approved','rework:BUILD','rework:SPEC')`; `('phase','phase_signoff')` → `('approved','changes')`; `('phase','escalation_tradeoff')` → `('resume','replan')`. Executors and dashboard consume the same object (Doctrine §9: one source); a test pins that the executors' accepted answers equal it.
- **`config.ModelRoute.tools: Literal['all','none'] = 'all'`** (additive amendment, CCR-class — ratified with this design) + adapters honor it (runner.py delta): the §4 tools-off spawn for `decision_session`. Default `'all'` preserves every existing route's behavior; the stub adapter ignores it.

## 7. Failure model (Doctrine §7)

| Failure | Handling |
|---|---|
| dashboard task crashes | `_dashboard_supervisor` contains it: 'alert' event + max-priority ntfy („Dashboard căzut — decizia pe telefon nu funcționează; fallback: cli decide"), restart after `dashboard.restart_delay_s`, forever (the founder channel must come back); paging deduped per §6 (first crash, then every Nth restart, counter in the payload); the supervisor itself **never lets any exception escape into the TaskGroup** — publish failures take the control-plane §6 `NotifyError` contract (`alert_delivery_failed` event, never re-raise) — so "**scheduler loop unaffected**" holds *because of* this containment, pinned by a unit test (failing publisher injected → loop keeps ticking) |
| bind/tailscale-IP resolution fails at startup | eager `DashboardServer.start()` in `cli run` **before** `run_forever` (§1/§6): `FactoryError` → orchestrator start aborts in the foreground with a clear message — never converted by the row-1 supervisor into an alert-and-retry loop running pipeline work founder-less; bind failures on supervised *restarts* are contained per row 1 |
| slow/hung client | per-connection socket timeout + thread-per-connection: a stuck phone connection ties up one daemon thread, never the loop, never other requests |
| render/handler error | **per-card containment first** (§2a): a card whose render raises becomes an explicit Romanian error card — one bad pending-decision row must never make every other decision unanswerable from the phone (DoD §13 „founder channel failing"); a page-level handler error → explicit 500 page in Romanian + stderr traceback — never a blank 200 (GETs don't write the DB, so no off-loop event rows) |
| `answer()` future timeout | 504 RO „reîncarcă" — retry safe (idempotent §3.1) |
| factory-repo commit race vs `cli decide` | index-lock collision: `GitError` → explicit 500; no partial state (file unregistered = inert). Clean interleaving (cli decide's whole commit+tx lands inside step 2's await window — the likelier case): caught at step 3's `status='pending'` guard → `ALREADY_ANSWERED` no-op page (§3.3), never a 500; D-0015 exception stays narrow |
| agent turn fails/times out **or is cancelled** in a session | `try/finally` teardown (§4): in-flight flag cleared + failed-turn notice appended on failure AND cancellation (a supervisor restart cancelling the turn must not wedge the session „busy" forever); explicit RO notice in the card; never auto-retried; option buttons unaffected |
| tailscale IP drifts/disappears at runtime | not silent-by-omission: the stale listener raises nothing, liveness stays fresh, yet `notify.dashboard_link` deep links (MagicDNS) resolve to the node's NEW IP → connection refused while everything reports green — contradicting the fail-loudly posture. The supervisor re-checks every `dashboard.bind_recheck_s`: re-run `tailscale ip -4` (or self-probe the bound socket) vs `bound_address`; mismatch/probe failure → 'alert' event + max-priority ntfy + restart `serve()` to re-resolve (rare — tailnet IPs are stable across daemon restarts — hence a cheap periodic check, not a watcher) |
| orchestrator down | page unreachable = honest signal; watchdog pages (DoD §9); `cli decide` is the emergency path (D-0015) |

## 11. v1.3 — Per-stage agent cost breakdown (founder directive 12-06-2026)

**Founder directive (verbatim intent):** for every stage running or finished, show the token cost per involved agent — "Agent N — (model) tokens in | tokens out / total USD for this agent, per the prices set in config" — both live ("la moment") and final ("la finalizare").

### 11.1 Sources of truth & the honesty rule

- **`token_ledger`** already records one row per agent run: `process_id, role, model, tokens_in, tokens_out, cost_usd, estimated, recorded_at`, keyed by `unit_id` — the per-stage, per-agent data EXISTS; this slice is read-path + config only. No schema change, no write-path change.
- **`cost_usd` precedence (Doctrine §21 — never show the founder false money):** claude routes report `total_cost_usd` from the CLI's own ndjson — exact and cache-aware (cache reads bill ~0.1×, writes ~1.25×; `tokens_in` is raw context flow, so a naive `tokens_in × price` would overstate real cost ~2-3×). Where `cost_usd` is non-NULL it renders as-is (exact). Where NULL (codex routes — model recorded as `default` — and stub), the dashboard ESTIMATES from config prices and renders with a `~` prefix.
- **NEW config section `pricing` (Doctrine §14):** `pricing.usd_per_mtok.<ledger-model>: {input: <usd>, output: <usd>}` — keys are LEDGER model strings (`fable`, `sonnet`, `haiku`, `default`, …), founder-tunable. Estimation: `tokens_in/1e6×input + tokens_out/1e6×output`. A NULL-cost row whose model has no pricing key renders `— (preț lipsă în config)` — fail-visible, never a silent zero (Doctrine §7). Rows with `estimated=1` token counts (bytes/4 fallback) keep the `~` regardless of cost source.
- Starter values (founder-tunable; API list prices 06-2026): fable 10/50, sonnet 3/15, haiku 1/5, opus-4-8 5/25; `default` (codex) 1.25/10 with a YAML comment marking it operator-estimated, to be verified against the OpenAI account — codex sums are `~` estimates either way. Costs are API-equivalent value (subscriptions pay differently); the legend says so.

### 11.2 UI (per the §10.2/§10.3 conventions — tables, phone-first, zero JS)

- **Plan section, per stage** (groups Finalizate + În lucru; Planificate have no ledger rows → nothing): under the phase table, one `<details>` per stage with ledger rows — summary line `«<etapă> — agenți: N · cost: <total>»` (running stages append „· în lucru — parțial"), body = one `<table>`: `agent (rol glossed R2) · model · tokeni intrare · tokeni ieșire · cost`. One row per ledger entry in `recorded_at` order (a re-run role appears twice — that is the truth of what was spent); total row last. The in-flight agent's run lands at its exit (ledger writes on finalize) — bounded staleness, stated in the legend.
- **„Etape în lucru" table (§10.3)** gains a right-aligned `cost` column after `tokeni` (same accrued-total semantics).
- **Phase header line** in the plan section gains the phase's total cost.
- **Totals with mixed precision** render exact and estimated SEPARATELY — `«12,40 $ + ~0,85 $»` — never silently summed into one figure.
- **Formatting:** new `_fmt_usd` — two decimals, Romanian decimal comma, `$` suffix with thin space; `<0,01 $` for sub-cent non-zero. New RO strings enter the `RO`/`GLOSS` tables (R1/R2; closure tests extend). Legend (small print, once per page, only when any cost cell rendered): „costurile = raportate exact de CLI (includ reducerile de cache); ~ = estimare din prețurile din config; sumele sunt echivalent-API (abonamentul se facturează separat); agentul în lucru apare la finalul rulării sale".
- No new inputs, no JS, no new write path — pure read-path rendering on the auto-refreshing page (the §10.9 no-free-text-input constraint is untouched).

### 11.3 Code deltas (CCR-10, control-plane v1.9→v1.10 annotation)
