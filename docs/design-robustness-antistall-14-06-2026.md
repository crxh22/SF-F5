NOTE: this design unifies the build program's task #3 (UNIT 1), #4 (UNIT 2), and #9 (UNIT 3 — the founder-flagged budget-reset notification) — they are ONE coherent program, not three separate efforts. Governing decisions: **D-0042** (escalation-routing founder DECISION: human stays arbiter, mechanical layer only, 30-min stuck-detector, architect-learns-≤5-min, persist-in-code), **D-0037** (capacity governor), **D-0041** (routing discussion). This program is the durable, in-orchestrator replacement for the **session-scoped bash monitor** the architect runs today (D-0041/D-0042: "the session-scoped monitor cârpă" → "the stuck-detector makes 'nothing sits silently >30min' a factory law"). All three units are orchestrator CODE that persists across every future session; all reuse the existing ntfy push (`notify.py`), the event log, and the periodic tick (`Scheduler._run`, scheduler.py:5009-5038).

## Roots being fixed

1. **Scheduler fairness (auth-access starvation class).** A unit in `ESCALATED` is category-`BLOCKED` (models.py:229,254); it re-enters dispatch only to run `_step_escalated` (scheduler.py:1333), which spawns NO agent (`_step_spawn_roles` returns `()` for ESCALATED — scheduler.py:1390-1391,1405). But the dispatch loop gates **every** unit-drive behind the agent-slot cap: `if len(self._tasks) >= cap: break` (scheduler.py:5112) fires BEFORE the step runs, so a resolved-but-not-yet-routed escalation (a control-plane-only pickup) waits for an agent slot it does not need, behind routine BUILD/SPEC/AUDIT agents. Same for gates and dispatch (PENDING→SPEC bookkeeping, MERGE_GATE Tier-1) — all `()`-spawn steps.

2. **Escalation routing + stuck escalations.** `escalations.target` is written by every creation site (nature/level: stage-conveyor→`phase_architect`, cross-cutting→`main_architect` — scheduler.py:1545,1661,2200,2374,3535,3596,3674,3767,3828,3868,3953,3980,3999,5171) but **no actor consumes it**. The dashboard renders it as VISUAL TEXT only (`escalation_reassurance: "în lucru la {target}; nu cere acțiunea ta"`, dashboard.py:154; glossed at 332-334). The only auto-resolver is the capacity governor, strictly scoped to limit-marked `agent_run_failed` (scheduler.py:1153-1216). The 24h `decision_latency_alert` (scheduler.py:5202) is for founder DECISIONS, not escalations. Net: a technical escalation pings the FOUNDER (wrong resolver), freezes the unit, and there is **no resolution-latency guarantee** — an escalation can sit `open` forever, and (the worse class) a **resolved escalation can sit un-advanced forever** if the stage never gets dispatched to pick it up (UNIT 1's exact failure) or if `_step_escalated` returns early (e.g. `last is None`, scheduler.py:2637 — operator surgery without a row). Both the incident-[20] resolved-not-advanced stall and the auth-access cap starvation are this class.

3. **Budget-reset → architect silence.** When the CapacityGovernor lifts a hold it emits `capacity_hold_ended` + pages the FOUNDER ("Capacitate revenită — fabrica a reluat singură", scheduler.py:1130-1134). The governor "never pages on entry" (scheduler.py:927) and on exit pages only the founder — the **architect** is not signalled and today learns the budget reset only by the founder relaying it (D-0042). The architect needs a durable, distinguishable signal to resume autonomously.

## Resolved decisions (decisive)

- **The notification target problem (cross-cutting, governs UNITs 2+3).** There is exactly ONE ntfy topic — the founder's (`claude-artur-md-hello`, factory.config.yaml:121; D-0004). There is no separate architect channel in code. The architect's real notification path today is the **session-scoped bash monitor polling the DB** (D-0041/D-0042). So "notify the architect" in code = (a) write a **distinct, machine-greppable event** the architect's monitor (and any successor's monitor) catches, AND (b) push ntfy with a **distinct title** so the founder relays correctly and a phone watcher can disambiguate. We do NOT invent an architect-only transport (Doctrine §0/§9 — no parallel infra; the founder explicitly accepted one shared topic, D-0004). The ≤5-min guarantee (D-0042) is met by the monitor's 45s poll over these events PLUS the ntfy as the human backstop. **This is the one place that needs an architect ruling — see Open Questions Q1.**
- **UNIT 1 fix = exempt no-spawn work from the cap, do NOT add aging.** The minimal, surgical change: in `_dispatch`, compute whether the unit's CURRENT step spawns an agent (reuse the EXISTING `_step_spawn_roles`, scheduler.py:1384/3489 — already the authoritative "does this step spawn" predicate, used by the capacity governor); only count cap-bounded slots for spawning steps; let `()`-spawn steps (dispatch, gates, ESCALATED) proceed past the cap. Aging/priority is rejected as over-engineering for this root (Doctrine §8): the starvation is specifically of zero-cost control-plane work, and exempting it is both necessary and sufficient. The invariant: **a no-spawn critical transition proceeds even when the cap is full of routine agents.**
- **UNIT 2 routing = a static `ESCALATION_ROUTE` table (token→actor+action), consumed by the detector and surfaced, NOT an autonomous resolver.** Founder hard terms (D-0042): human stays arbiter, NO resolver agent, do not automate judgment. So "live routing" makes `target` DRIVE the stuck-detector's escalate-UP ladder and the notification recipient — it does not auto-resolve. The actor ladder is `phase_architect → main_architect → founder` (the existing target vocabulary + the founder as the top product authority).
- **UNIT 2 stuck-detector runs on the periodic tick**, alongside `_stall_detector` and `_decision_latency_alerts` (scheduler.py:5022-5024) — same loop body, same `_notify`, same event-log + delivery-failure-dedup pattern. It catches BOTH (a) `open` > threshold and (b) **resolved-but-stage-not-advanced** > threshold. Detecting (b): an escalation `status='resolved'` whose `resolved_at` is older than the threshold AND whose unit is STILL in `ESCALATED` (the resolution never got picked up — exactly UNIT 1's starvation and the incident-[20] class). This is a pure DB predicate (Doctrine §20), no agent.
- **UNIT 2 escalate-UP = bump the OPEN escalation's `target` one rung + re-page**, and for the resolved-not-advanced case, **re-page + alert** (the row is already resolved; the fix is dispatch, which UNIT 1 delivers — the detector's job is to make the silence LOUD, per D-0042 "nothing sits silently >30min"). It must NOT auto-resolve or auto-create a new escalation that could loop. Threshold = configurable, default 30 min (D-0042 `T_architect`).
- **UNIT 3 = page the architect on `capacity_hold_ended`** by reusing UNIT 2's architect-notification path (the distinct event + distinct ntfy title). Smallest possible: ONE extra publish (distinct title) inside the existing `_lift_hold` (scheduler.py:1114-1151), guarded exactly like the existing founder page (NotifyError → `alert_delivery_failed`, never raise). The `capacity_hold_ended` EVENT already exists (scheduler.py:1119-1126) and is already greppable by a watcher — so the durable machine signal is ALREADY THERE; UNIT 3 adds the architect-addressed ntfy + (optionally) a distinguishing payload key. Relationship to UNIT 2: same notification helper, so build UNIT 2 first.

## UNIT 1 — scheduler fairness (no-spawn work exempt from the agent-slot cap)

### Root
`_dispatch` (scheduler.py:5074) breaks out of the dispatch scan at `if len(self._tasks) >= cap: break` (5112) for the FIRST eligible unit once the cap is reached — regardless of whether that unit's step needs an agent. A unit in `ESCALATED` with a freshly-resolved escalation is category-`BLOCKED`, becomes `eligible` because its open-escalation count dropped (5100-5109), but is then blocked by the cap from running `_step_escalated`, which spawns nothing. A critical stage's resolution waits behind routine agents.

### Resolved decisions
- The "does the current step spawn an agent?" question is ALREADY answered by `_step_spawn_roles` (scheduler.py:1384 stage, 3489 phase): it returns the role tuple a step may spawn, `()` for dispatch/gates/ESCALATED. The capacity governor already relies on this exact contract (scheduler.py:1340-1342). REUSE it — do not re-derive the no-spawn set (single source, Doctrine §0/§9).
- Cap-accounting changes from "≥ cap → stop dispatching anything" to "≥ cap → stop dispatching SPAWNING steps; no-spawn steps still dispatch." A no-spawn drive occupies a `self._tasks` slot only for the duration of one control-plane step (no subprocess), so it does not meaningfully compete for the economic resource the cap protects (the §7 agent-process budget).
- The cap counts SPAWNING tasks only. Because no-spawn drives are brief and never hold an LLM, counting them against the cap is what causes the starvation; not counting them is the fix.

### Per-file changes
- **`scheduler.py` `UnitExecutor` Protocol (1245-1254) + both executors** expose a public spawn-predicate so the Scheduler can ask the executor about a unit it is about to dispatch. Today `_step_spawn_roles` is a private method on `StageExecutor` (1384) and `PhaseExecutor` (3489) keyed off `unit.state`; the Scheduler holds `self._executors: Mapping[Level, UnitExecutor]` (4611, 4627) and drives via `self._executors[level].execute(unit_id)` (5122). The Protocol is minimal (`level`, `execute(unit_id)` — 1248,1250). Two viable shapes:
  - **(A)** add `spawn_roles(self, unit) -> tuple[str, ...]` to the Protocol + a one-line public wrapper on each executor (`return self._step_spawn_roles(unit)`). The Scheduler then needs the UNIT OBJECT (not just `unit_id`) to call it — couples with the `_scan_units` carry-the-unit change below.
  - **(B)** add `step_spawns_agent(self, unit_id: str) -> bool` to the Protocol; each executor re-reads its own unit (`self._stage(unit_id)`/`self._phase(unit_id)`, 1361/3471) and returns `bool(self._step_spawn_roles(unit))`. Keeps the Protocol on `unit_id` (matching `execute`), no `_scan_units` change, at the cost of one extra `get_stage`/`get_phase` per cap-relevant candidate per tick.
  Either is ~4 lines/executor + 1 Protocol line. Recommend (A) with the `_scan_units` carry (one read, both consumers).
- **`scheduler.py` `_dispatch` (5074-5116)** — replace the single cap gate with a spawn-aware one. Concretely, after `eligible` is computed and before `self._tasks[key] = ...` (around 5110-5114):
  - fetch the unit (the executor needs the unit object; `_dispatch` currently works off `(level, unit_id, category)` from `scan` — read the unit via `fdb.list_units`/`get_stage`/`get_phase` once, or have `_scan_units` carry the unit object alongside the category to avoid a re-read);
  - `spawns = bool(self._executors[level].spawn_roles(unit))`;
  - change `if len(self._tasks) >= cap: break` to:
    ```
    if spawns and self._spawning_task_count() >= cap:
        continue   # a SPAWNING step waits for a free slot; keep scanning for no-spawn work
    ```
    Note `continue`, not `break`: a later no-spawn unit in the scan must still be reachable (a `break` would strand it behind a capped spawning unit).
  - track which dispatched tasks are spawning: keep a `self._spawning: set[tuple[Level,str]]` updated when a spawning task is created and discarded in `_reap`/`_drive.finally`; `_spawning_task_count()` = `len(self._spawning)`. (Simpler alternative: recompute spawn-ness of each live `self._tasks` key on demand — but the unit's state can change under a running task, so the snapshot-at-dispatch set is the honest count.)
- **`scheduler.py` `_scan_units` (5047-5072)** — OPTIONAL: extend the tuple to `(level, unit_id, category, unit)` so `_dispatch` does not re-query each unit. Keeps the change localized; `_stall_detector` (5253) and `_dispatch` both consume `scan` — update both unpackings. (If you prefer minimal blast radius, skip this and have `_dispatch` re-read the unit only for the not-yet-in-`_tasks`, eligible, cap-relevant candidates.)

### Key tests (incl. the invariant pin)
- **THE INVARIANT PIN** — `test_no_spawn_escalation_proceeds_when_cap_full`: set `max_parallel_agents=1` (or 2), fill the cap with long-running SPAWNING stages (ScriptedExecutor stages stuck in BUILD), put one stage in `ESCALATED` with a resolved escalation (`rework:VALIDATE`); drive ticks; assert the ESCALATED stage transitions OUT of ESCALATED (its `_step_escalated` ran) **while the cap is full**. Mutation check: without the fix the ESCALATED stage stays put until a spawning slot frees. (Mirror `test_parallel_cap_bounds_concurrent_units`, tests/unit/test_scheduler.py:448; reuse `make_scheduler`/`ScriptedExecutor`/`run_blocked`.)
- `test_spawning_steps_still_capped`: N+1 stages all in a SPAWNING state, cap=N → exactly N concurrent agent spawns, the (N+1)th waits (the cap STILL bounds real agents — the existing `test_parallel_cap_bounds_concurrent_units` invariant must stay green; do not weaken it).
- `test_no_spawn_does_not_starve_later_no_spawn` (the `continue`-not-`break` pin): cap full of spawners, TWO no-spawn ESCALATED units later in the scan → BOTH proceed in the same tick.

### Risks / foot-guns
1. `_step_spawn_roles` is keyed off the unit's CURRENT state; a unit dispatched as no-spawn (ESCALATED) whose `_step_escalated` routes it to BUILD will spawn on its NEXT dispatch — that next dispatch is correctly cap-gated. No double-count if the spawning-set is snapshot at dispatch and cleared on drive completion.
2. The conservative reading of `_step_spawn_roles` (AUDIT/VALIDATE return non-empty even for the conditional CP-1/responder, scheduler.py:1386-1388) is CORRECT here too — those steps CAN spawn, so they stay capped. Do not narrow it.
3. Do NOT let a flood of no-spawn drives starve the loop: they are brief and bounded by the number of ESCALATED/gate units; acceptable. (Falsifiable below.)
4. The phase-level `_step_spawn_roles` (3489) must be exposed identically — a phase in ESCALATED has the same starvation.

### Falsifiability (Doctrine §10)
We'll know it's wrong if: a resolved escalation on a critical stage STILL waits behind routine agents when the cap is full (the fix didn't take); OR real concurrent agent spawns exceed `max_parallel_agents` (the cap was over-loosened — the spawn-ness predicate is wrong); OR the loop tick latency spikes because no-spawn drives are unbounded (they should be O(ESCALATED+gates), tiny).

**Amendment after build (the achievable invariant).** The build revealed that `execute()` walks every legal step in ONE asyncio task, so the originally-stated "real concurrent agents NEVER exceed `max_parallel_agents`" does not hold as an absolute: an exempt BLOCKED drive whose resolution routes to a rework SPAWNING state (e.g. ESCALATED→`rework:BUILD`) walks straight from the no-spawn pickup INTO the rework spawn in the same task, spawning once past the cap (it is never added to `self._spawning`). The achievable invariant is therefore **steady-state concurrency ≤ cap; a transient ≤ cap+K overshoot** occurs when K rework-routing escalations resolve in one tick (K bounded by simultaneously-resolved rework escalations ≤ cap; the capacity governor's batch auto-resolve at a budget reset is the K≈cap case). This is ACCEPTED as an economic-cap tolerance — the cap protects the §7 process budget, not a hard safety limit (§8: no heavier mechanism without an observed incident) — and is pinned by `test_rework_routing_overshoots_cap_by_bounded_k_accepted_residual` (cap=2, K=3 → measured peak 5; asserts `cap < peak ≤ cap+K`, so any worsening toward unbounded is caught and the residual stays a visible, tested fact). **Revisit trigger:** if budget-reset oscillation is ever observed (capacity hold→resume→re-hold churn), add a spawn-point `asyncio.Semaphore(max_parallel_agents)` in `_run_step_agent` to make the cap hard (the fix deferred per Doctrine §8/§12).

## UNIT 2 — escalation routing + stuck-escalation detector (D-0042, founder-approved mechanical layer)

### Root
`escalations.target` is dead weight — written everywhere, consumed by no actor (only rendered, dashboard.py:154,332-334). No resolution-latency guarantee exists for escalations (the 24h alert is decision-only, scheduler.py:5202). An escalation can sit `open` indefinitely, and a resolved escalation can sit un-advanced indefinitely. Founder approved (D-0042) ONLY the mechanical layer: live routing + stuck-detector + richer actions; NO autonomous resolver.

### Resolved decisions
- **Live routing = a static `ESCALATION_ROUTE` ladder in `models.py`** (the `STAGE_ESCALATION_RESOLUTIONS`/`GATE_ANSWERS` precedent — one source, consumed by scheduler + dashboard). Two facets it drives: (1) the **escalate-UP rung** for the stuck-detector; (2) the **recipient** of the page. Shape:
  ```
  ESCALATION_TARGET_LADDER = ("phase_architect", "main_architect", "founder")
  ```
  `target` already takes the first two values at creation; `founder` is the top rung. "Action per level" (D-0042) is the EXISTING resolution vocabulary scoped by who may apply it — documented, not a new code path (the founder ruling forbids auto-action). The ladder's only mechanical effect is the stuck-detector's bump + recipient choice.
- **Stuck-detector on the tick** — a new `async def _stuck_escalation_detector(self, scan)` called in `_run` right after `_decision_latency_alerts()` (scheduler.py:5024). Two predicates, both pure DB:
  - **(a) open-too-long:** an escalation `status='open'` with `created_at` older than `escalation.stuck_escalation_threshold_min` AND not yet bumped this episode → bump `target` one rung UP the ladder + re-page (recipient = the NEW rung) + `escalation_bumped` event. De-dup with a per-escalation latch (mirror `mark_decision_alerted`, scheduler.py:5239, OR an in-memory `set` like the stall latch, scheduler.py:5274-5275) so it fires once per rung, not every tick.
  - **(b) resolved-but-not-advanced:** an escalation `status='resolved'` with `resolved_at` older than the threshold AND its unit STILL in `ESCALATED` (read state via `get_stage`/`get_phase`; the resolution never got picked up). → page (recipient = `target`/architect) + `escalation_stuck_resolved` event, ONCE per episode. This is the case that catches incident-[20] (resolved, stage never advanced) AND the auth-access cap-starvation symptom (UNIT 1 fixes the cause; this is the backstop). It does NOT auto-resolve or re-create — the row is already resolved; the silence is what's wrong (D-0042).
- **The ≤5-min architect guarantee (D-0042 HARD)** is met by the distinct EVENTS the architect's monitor polls (45s) + the ntfy. So the detector writes `escalation_bumped`/`escalation_stuck_resolved` events AND calls the shared **`_notify_architect`** helper (below). The HARD requirement is about the architect LEARNING of an escalation — so the detector also covers the *initial* notification gap: today escalation creation pages only the founder/dashboard via the trade-off card path; the architect learns only via the monitor. **Decision:** the detector is the durable architect-notification mechanism for ALL open escalations targeting an architect — on the FIRST tick an architect-targeted escalation is seen open and un-notified (age 0, before any threshold), emit one `escalation_opened_notice` event + `_notify_architect` page, then the 30-min bump is the escalate-UP. This makes "architect learns within ≤5 min" a code law independent of the session monitor. (If the architect prefers to keep first-notification in the session monitor and use the orchestrator only for the >30-min bump, that is Open Question Q2.)
- **`_notify_architect(title, link)` helper** — reuses `self._notify.publish` with a DISTINCT title prefix (e.g. `"[arhitect] …"`) and the existing NotifyError→`alert_delivery_failed` dedup contract (mirror `_decision_latency_alerts`, scheduler.py:5210-5239). Same ntfy topic (only one exists); the distinct title lets the founder relay correctly and a phone watcher disambiguate. The EVENT is the machine signal the monitor greps. UNIT 3 reuses this helper.
- **Richer resolution actions (D-0042 (c))** — DEFER beyond documenting the ladder, UNLESS the architect wants the "re-run only the failed gate" action now. `rework:MERGE_GATE` already IS the "re-run only the merge gate" action (D-0042, models.py:294). No other granular re-entry has a fired incident (Doctrine §8). **Open Question Q3.**

2. **The "open" age clock** uses `created_at` (escalation insert time) — correct for "open too long". The "resolved-not-advanced" clock uses `resolved_at`. Don't confuse them.
3. **`founder` as a `target` rung** must not break the dashboard gloss (it already glosses `founder`, dashboard.py:334) or the trade-off card path (founder-targeted escalations already surface as decision cards when they reach AWAITING_HUMAN — but the stuck-detector's bump to `founder` does NOT transition the unit to AWAITING_HUMAN; it only re-labels + pages). Be explicit: bumping `target` is a LABEL+RECIPIENT change, never a state transition (the founder ruling: no auto-action). If the architect wants a `founder` bump to also raise a decision card, that's judgment-adjacent — leave to the architect (Q3).
4. **The ≤5-min guarantee depends on the architect's monitor polling these events.** The orchestrator emits them durably; the monitor must grep `escalation_opened_notice|escalation_bumped|escalation_stuck_resolved`. Document this in session-succession.md / architect-operations.md so every successor's monitor watches them. (The ntfy is the human backstop if the monitor is down.)
5. Do NOT let the detector resolve or transition anything — it is detection + notification + a target relabel ONLY (Doctrine §20; the founder's NO-resolver-agent line).

### Falsifiability (Doctrine §10)
We'll know it's wrong if: an escalation sits `open` or `resolved-not-advanced` longer than the threshold without a single page or event (the detector didn't fire); OR the detector pages every tick (the latch is broken — alarm fatigue degrades the §9 channel); OR it ever flips `status` / transitions a unit / spawns an agent (it overstepped the founder's mechanical-only mandate); OR the architect's monitor sees no new event within one poll of the threshold crossing.
